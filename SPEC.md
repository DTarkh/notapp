# Notapp ‚Äî Technical Specification

## 1. Project Overview

**Notapp** is a web-based note-taking application that allows authenticated users to create, manage, and optionally share rich-text notes. Notes are private by default and can be made publicly accessible via a shareable link.

## 3. Authentication

### 3.1 Providers

- **Email + Password** ‚Äî standard credential flow with email verification
- **Google OAuth** ‚Äî via better-auth's Google provider

### 3.2 Session Management

- Sessions managed by better-auth (server-side sessions stored in the database)
- Session cookie set on login; cleared on logout
- Protected routes redirect to `/auth` if no valid session exists

### 3.3 Routes

| Route            | Access                                                   |
| ---------------- | -------------------------------------------------------- |
| `/auth`          | Public ‚Äî single page with login and register tabs/toggle |
| `/auth/callback` | Public (OAuth redirect handler)                          |
| All other routes | Authenticated only                                       |

### 4.2 `notes` table

Defined manually in Drizzle and migrated with `drizzle-kit`.

| Column        | Type                  | Notes                                                        |
| ------------- | --------------------- | ------------------------------------------------------------ |
| `id`          | `text` (PK)           | UUID, generated by app                                       |
| `user_id`     | `text` (FK ‚Üí user.id) | Owner; cascade delete                                        |
| `title`       | `text`                | Derived from first heading or first line of content          |
| `content`     | `text`                | TipTap JSON serialised as string (or `jsonb`)                |
| `is_public`   | `boolean`             | Default `false`                                              |
| `public_slug` | `text`                | Unique short random slug; nullable; generated on first share |
| `created_at`  | `timestamp`           | Default `now()`                                              |
| `updated_at`  | `timestamp`           | Updated on every save                                        |

**Indexes:**

- `(user_id, created_at)` ‚Äî for sorted note list queries
- `(public_slug)` ‚Äî unique, for public note lookup
- Full-text search index on `(title, content)` ‚Äî via PostgreSQL `tsvector` or Supabase full-text search

## 5. Features & Functional Requirements

### 5.1 Note CRUD

| Action     | Details                                                                                                                                                                                            |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Create** | Opens editor with blank state. Title auto-derived from first heading or first non-empty line. Note saved on explicit user action (Save button) or auto-save (debounced, ~2s after last keystroke). |
| **View**   | Renders TipTap content as read-only when not in edit mode. Full note page at `/notes/:id`.                                                                                                         |
| **Update** | Same editor, same auto-save behaviour. `updated_at` timestamp refreshed on every save.                                                                                                             |
| **Delete** | Confirmation dialog before deletion. Permanent ‚Äî no soft delete.                                                                                                                                   |

### 5.2 Note List

- Displayed on the main dashboard (`/`)
- Each note card shows: title, short content preview, `created_at` date
- **Sort:** by date added (newest first by default, toggle to oldest first)
- **Search:** full-text search across title and content, client-triggered (search input with debounce), results filtered server-side

### 5.3 Public Sharing

- Each note has a toggle to make it public or private
- On first share, a unique short random slug is generated and stored (e.g. `xk29a`). The slug persists even if the note is unshared and re-shared later
- **Public URL:** `/s/:slug` ‚Äî accessible without authentication
- Public view is **read-only** and renders note content with no UI chrome (no edit controls, no auth required)
- Unsharing sets `is_public = false`; the slug is retained but the `/s/:slug` route returns 404

### 5.4 Rich Text Editor (TipTap)

**Enabled extensions:**

| Feature            | TipTap Extension                                             |
| ------------------ | ------------------------------------------------------------ |
| Bold               | `Bold`                                                       |
| Italic             | `Italic`                                                     |
| Heading levels 1‚Äì3 | `Heading` (levels: [1, 2, 3])                                |
| Normal paragraph   | `Paragraph` (default)                                        |
| Inline code        | `Code`                                                       |
| Code block         | `CodeBlock` (or `CodeBlockLowlight` for syntax highlighting) |
| Bullet list        | `BulletList` + `ListItem`                                    |
| Horizontal rule    | `HorizontalRule`                                             |
| History            | `History` (undo/redo)                                        |

**Toolbar controls:** Bold, Italic, Heading dropdown (H1/H2/H3/Normal), Inline Code, Code Block, Bullet List, Horizontal Rule, Undo, Redo.

**Storage format:** TipTap JSON (`editor.getJSON()`) serialised and stored as `jsonb` in PostgreSQL. Loaded back via `editor.commands.setContent()`.

## 7. API / Server Functions

TanStack Start server functions (`createServerFn`) will be used for all data access. No separate REST API layer.

| Server Function                              | Description                                      |
| -------------------------------------------- | ------------------------------------------------ |
| `getNotes({ userId, search?, sort })`        | Returns user's notes, filtered and sorted        |
| `getNote({ id, userId })`                    | Returns single note by ID (must belong to user)  |
| `createNote({ userId, title, content })`     | Inserts new note, returns created note           |
| `updateNote({ id, userId, title, content })` | Updates note content and `updated_at`            |
| `deleteNote({ id, userId })`                 | Deletes note (validates ownership)               |
| `toggleShare({ id, userId })`                | Flips `is_public`; generates slug if not yet set |
| `getPublicNote({ slug })`                    | Returns note by slug if `is_public = true`       |

All mutating functions validate that the requesting user owns the resource before executing.

## 8. Styling & Design System

Styles are written in **CSS Modules** (`.module.css` per component). Global design tokens are defined as **CSS custom properties** on `:root` in `global.css`, overridden under `[data-theme="dark"]`. ### 8.7 React Aria & Component Conventions

React Aria components are **unstyled by default** ‚Äî all visual styling is applied via CSS Modules using the `className` prop. React Aria exposes data attributes for interactive states that are used as CSS selectors instead of manually managing state classes.

**Key data attributes used in styles:**

```css
[data-hovered] {
  background: var(--color-bg-subtle);
}
[data-pressed] {
  opacity: 0.85;
}
[data-focused] {
  outline: 2px solid var(--color-accent);
  outline-offset: 2px;
}
[data-disabled] {
  opacity: 0.4;
  cursor: not-allowed;
}
[data-selected] {
  background: var(--color-accent);
  color: #fff;
} /* ToggleButton active */
[data-invalid] {
  border-color: var(--color-danger);
}
```

**General rules:**

- Buttons use `--radius-sm` (2px) ‚Äî squared-off, technical feel
- Inputs use `--radius-sm` with a `1px` border; `[data-focused]` swaps border to `--color-accent`
- No drop shadows ‚Äî depth expressed through borders and background contrast
- Dialogs (delete confirmation) use `--radius-md`, `1px` border, and a dim backdrop
- TipTap toolbar uses React Aria `ToggleButton` components; active format indicated via `[data-selected]`
- Each React Aria component gets its own CSS Module (e.g. `Button.module.css`, `TextField.module.css`) in `app/components/ui/`

## 11. Key Implementation Notes

### Slug Generation

Use `nanoid` with a custom alphabet (alphanumeric, no ambiguous chars) at length 8‚Äì10. Generate on first share, never regenerate.

### Auto-Save

Debounce TipTap `onUpdate` callback by ~2000ms. Show a "Saving‚Ä¶" / "Saved" status indicator in the toolbar. Use an `isDirty` flag to avoid redundant saves.

### Title Derivation

On every save, derive the note title server-side by extracting the text content of the first non-empty `heading` or `paragraph` node from the TipTap JSON. Truncate to 100 characters.

### Search Implementation

Use PostgreSQL full-text search via a `tsvector` generated column on `(title || ' ' || content)`. Query via `plainto_tsquery`. Alternatively, use `ILIKE` for simplicity in v1.

### Public Route Security

The `/s/:slug` route must **only** return notes where `is_public = true`. It must not expose the `user_id` or any user data in the response.

### Ownership Validation

Every note server function that reads or mutates a specific note must verify `note.user_id === session.user.id` and throw a 403 otherwise.
